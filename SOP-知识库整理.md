# 颖响力知识库 SOP

## 目标

这套 SOP 的目标不是“单篇写一篇文稿”，而是把每一期节目拆进一个可持续扩展的网页知识库里。最终产物包含两层：

- 内容层：节目、概念、思想模型、人物、主题、关键词
- 展示层：本地静态网页，`localhost` 可打开

你以后打开一个新终端，只需要传入 `.srt` 路径，就能先把新节目自动收录到这套知识库结构中，再继续补内容。

## 内容设计原则

### 1. 节目是入口，不是终点

每个节目 JSON 负责承接当期内容，但真正长期可复用的是：

- 概念
- 思想模型
- 人物
- 主题
- 关键词

节目要不断把内容往这些稳定节点上沉淀。

### 2. 信息、概念、思想模型是点线面关系

- 信息：字幕里的具体材料、表达、细节
- 概念：比思想模型小，概括一个现象
- 思想模型：多个概念按机制或因果关系组织成更高层框架

### 3. 思想模型优先复用 GV 现有模型

如果 GV 已经有贴合模型，优先复用模型名，再结合本期解释。
只有当 GV 现有模型明显不贴合时，才新增模型。

## 节目条目字段规范

每个节目 JSON 至少包含：

- `id`
- `title`
- `summary`
- `videoLinks`（可选）
- `tags`
- `people`
- `themes`
- `concepts`
- `models`
- `relatedEpisodes`
- `topic`
- `viewpoints`
- `extensions`

其中：

- `videoLinks`
  可选。用于在节目详情页标题下方显示视频平台图标。
  每个条目推荐包含：
  - `platform`: `bilibili` 或 `youtube`
  - `url`: 视频链接
  - `access`: 可选，普通节目可省略；如果是会员专属内容，填 `member`
  - `status`: 可选，如果平台没有这一期，填 `unavailable`
  - `note`: 可选，用于鼠标移到灰色按钮时显示，例如 `已下架`

- `topic`
  必须按这套结构写：
  - `background`
  - `conflicts`
  - `boundaries`
  - `mechanism`
  - `extensions`
- `viewpoints`
  必须写成分小标题的观点数组

## 六类内容怎么写

### 节目 `content/episodes/*.json`

负责承接本期的完整结构化整理，是网页详情页的主来源。

如果已经找到视频链接，优先直接写进节目 JSON 的 `videoLinks`，不要拖到最后再补。

示例：

```json
"videoLinks": [
  {
    "platform": "bilibili",
    "url": "https://www.bilibili.com/video/..."
  },
  {
    "platform": "youtube",
    "url": "https://www.youtube.com/watch?v=..."
  }
]
```

如果 B 站是会员专属节目，写成：

```json
"videoLinks": [
  {
    "platform": "bilibili",
    "url": "https://www.bilibili.com/video/...",
    "access": "member"
  }
]
```

前端会自动把普通 B 站图标显示成蓝色，把会员专属 B 站图标显示成金色会员标识。

如果 B 站没有这一期，写成：

```json
"videoLinks": [
  {
    "platform": "bilibili",
    "status": "unavailable",
    "note": "已下架"
  }
]
```

前端会自动把这个入口显示成灰色，鼠标移上去显示 `已下架`。

## 视频链接维护标准

视频链接现在是节目页标配，不是可有可无的附加信息。

默认要求：

1. 每期节目都维护 `videoLinks`
2. 优先同时维护两条：
   - `bilibili`
   - `youtube`
3. 如果用户没有单独提供对照视频链接，必须由整理者自己主动搜索
   - 先从官方频道首页 / 投稿页直接找
   - B 站优先看 UP 主页与 `投稿`
   - YouTube 优先看频道首页与 `videos`
   - 搜索时先用节目主标题，不要一上来只搜零散关键词
   - 再用 `｜` 后面的关键词尾巴组合搜索
   - B 站和 YouTube 都要各自至少搜一轮
   - 不允许因为“用户没给链接”就直接留空
   - 也不允许在未主动搜索前就先写 `unavailable`
   - 如果 B 站默认投稿流没找到，还必须继续检查 `充电专属` 或标题里带“第二部分内容请充电收看”的相关视频
4. B 站分三种状态：
   - 普通节目：蓝色
   - 会员节目：金色会员样式
   - 没有 / 已下架：灰色，并在 hover 时显示 `已下架`
5. YouTube 默认正常可用，当前项目按全量可用维护

批量同步命令：

```bash
npm run sync:videos
```

这个命令会自动：

1. 读取本地 B 站库存与当前清单
2. 抓取 YouTube 频道视频列表
3. 批量回填 `content/episodes/*.json` 里的 `videoLinks`
4. 自动执行 `npm run build`

也就是说，新终端里按 SOP 执行到这里时，不需要手工一个个点视频链接。

### 当自动同步找不到时怎么办

如果 `npm run sync:videos` 没有命中某一期：

1. 不要立刻把“找不到”当成最终结果
2. 必须手工去两个平台各搜一轮：
   - 先查官方频道首页 / 投稿页
   - 再搜主标题
   - 再搜主标题里的核心名词
   - 再搜 `｜` 后面的关键词尾巴
3. 手工确认后，再决定写成：
   - 真实 `url`
   - `access: "member"`
   - 或最终 `status: "unavailable"`
4. 只有在自动同步、官方频道页检查和手工关键词搜索都失败后，才允许把该期标成 `unavailable`

### 视频链接判别标准

#### B 站怎么判别

B 站不是直接硬抓网页前台结果，而是优先使用本地已经保存的清单文件。

主要来源：

- `../bilibili/raw/subtitle_inventory.json`
- `../bilibili/raw/season_episodes.json`
- `scripts/video-link-overrides.json`

判别顺序固定为：

1. 先读 `scripts/video-link-overrides.json`
   - 这里放人工确认过的链接
   - 只要覆盖表里有，脚本就直接信这个值
   - 后续批量同步不得覆盖人工确认链接
2. 再读本地 B 站库存：
   - `subtitle_inventory.json`
   - `season_episodes.json`
3. 用 `EPxxx` 编号优先匹配
4. 如果标题里没有明确 `EPxxx`，再用归一化标题做模糊匹配
5. 如果本地 B 站清单里确实找不到，才判成：
   - `status: "unavailable"`
   - `note: "已下架"`

这一条要特别强调：

- 只要 `video-link-overrides.json`
- 或 `subtitle_inventory.json`
- 或 `season_episodes.json`

三者里任意一个能证明这一期在 B 站存在，就**禁止**把它标成灰色 `已下架`。

也就是说：

- “没在公开页看到” 不等于已下架
- “其实在会员里” 也不等于已下架
- 必须在本地证据链也找不到，才能判灰

#### B 站会员怎么判别

B 站会员态不是靠颜色猜，是靠数据或明确文本线索判断。

目前判定依据：

- `season_episodes.json` 里的 `attribute === 8`
- 标题或分 P 名里明确出现：
  - `会员`
  - `会员专属`
  - `搬后会员发`
- 人工覆盖表里显式写：
  - `"access": "member"`

满足以上任一条，就按会员节目处理。

#### YouTube 怎么抓

YouTube 不是手工搜每一条，而是直接抓频道数据。

来源：

- 频道页：`https://www.youtube.com/@颖响力/videos`

抓取方式：

1. 先从频道页 HTML 里提取：
   - `ytInitialData`
   - `INNERTUBE_API_KEY`
   - `INNERTUBE_CLIENT_VERSION`
   - `VISITOR_DATA`
2. 先读首页首屏视频
3. 再用 YouTube 内部 `browse` continuation 接口继续翻页
4. 直到把频道视频列表抓全

也就是说，YouTube 当前不是靠页面可见的 30 条，而是靠 continuation 翻完整个频道。

#### YouTube 怎么匹配到 EP

优先级：

1. 标题中的 `EPxxx`
2. 描述摘要中的 `EPxxx`
3. 兼容旧写法：
   - `PE21` 这种也会识别成 `EP021`
4. 如果标题不带编号，再用归一化标题做模糊匹配

#### 手工覆盖优先

凡是你单独给过我的视频链接，都要写进：

- `scripts/video-link-overrides.json`

这是最高优先级来源。

规则：

- 手工覆盖优先于自动抓取
- 后续再跑 `npm run sync:videos` 也不能把手工链接冲掉
- 会员节目如果已经人工确认，直接在覆盖表里写 `access: "member"`
- 如果一条节目已经被人工从灰色修正为可用或会员，后续同步绝不能再回退成灰色

#### 新终端执行原则

新终端如果要补视频链接，不要自己重新发明流程。

必须按下面顺序做：

1. 先看 SOP 这段视频链接规则
2. 先看 `scripts/video-link-overrides.json` 有没有人工确认链接
3. 再运行：

```bash
npm run sync:videos
```

4. 跑完以后抽查三种样本：
   - 一条普通 B 站
   - 一条会员 B 站
   - 一条灰色已下架 B 站
   - 以及一条刚被人工修正过的节目，确认没有被重新打回灰色
5. 最后再看本地页面

最低抽查标准：

- 普通：蓝色 B 站入口
- 会员：金色 B 站入口
- 下架：灰色入口，hover 显示 `已下架`
- 人工修正过的节目：必须保持修正后的状态，不能被脚本回退

每个概念都要能单独成立，最少要有：

- `name`
- `summary`
- `definition`
- `importance`
- `episodes`

### 思想模型 `content/models/*.json`

每个思想模型最少要有：

- `name`
- `summary`
- `definition`
- `application`
- `sourcePath`
- `episodes`

### 人物 `content/people/*.json`

人物不是简单标签，必须有最小介绍。注意：`content/people` 是源数据层，不再作为前台独立入口，站点会把人物并入关键词页。

- `summary`
- `description`
- `episodes`

要求：

- `summary` 先回答“这个人是谁”
- `description` 先补这个人的背景，再写“在这些节目里主要怎么被提及”
- 外国人物补 `englishName`
- 如果改过旧 slug，旧写法放进 `aliases`
- 不能只写标签式名字，必须有一句到一段可直接展示在网页上的简单背景介绍
- 不要让 `description` 一上来就是：
  - `在颖响力知识库里……`
  - `为什么重要……`
- 人物源数据的优先顺序是：
  1. 这个人是谁
  2. 他/她在这些节目里主要承接什么讨论线

### 主题 `content/themes/*.json`

主题用于把多个节目按议题聚合，最少要有：

- `summary`
- `description`
- `episodes`

### 关键词 `content/keywords/*.json`

关键词层用于承接品牌、人物、产品、公司、实体等更具体的检索入口，类似 GV 的公司层。人物对外统一从这里进入，不再保留独立人物页体系。

最少要有：

- `summary`
- `description`
- `aliases`
- `relatedKeywords`
- `episodes`

要求：

- `summary` 先说明这个关键词是什么类型的对象，例如人物、公司、品牌、产品、实体入口
- `description` 必须先写“它是什么 / 做什么”，再写“它为什么在颖响力知识库里重要”
- 如果这个关键词对应某个人物，优先复用 `content/people` 的信息，不要再手写一份相互冲突的人物介绍
- `relatedKeywords` 用来放：
  - 近义但不应并入 `aliases` 的相近词
  - 上位词 / 下位词
  - 同一议题簇里需要互相跳转的相关词

例如：

- 人物关键词：先写这个人是谁，再写其在专题里的位置
- 公司关键词：先写这家公司做什么，再写它在节目里代表的结构问题

## 节点生成硬标准

这一节不是写作建议，而是实际落库时必须遵守的标准。

### 1. 新增节点前，先查有没有现成节点

不要看到一个新说法就立刻新建 `concept` 或 `model`。

这是默认规则，不是可选动作。

先做两步搜索：

```bash
rg -n "咽喉|命门|海峡|中立" content/concepts content/models
rg -n "英文 slug|拼音 slug|中文近义词|英文常用名" content/concepts content/models content/themes content/people content/keywords
```

判断顺序：

1. 如果只是同一机制的另一种说法，优先复用已有节点
2. 如果只是当前节目的一次性表达，先留在 `summary / topic / viewpoints`
3. 只有当它能跨节目复用、且现有节点明显装不下时，才新增节点

复用已有节点的收益要写死理解：

- 省时间，不重复造轮子
- 提高整理效率，不用每期都发明新概念、新模型
- 让原有节点越用越厚，`episodes / context / related*` 会越来越完整
- 让知识库入口更稳定，读者更容易形成连续认知，不会被一堆近义新词打散

所以默认动作永远是：

`先查旧节点 -> 能复用就复用 -> 复用后把旧节点补厚`

而不是：

`先新建一个差不多的新节点 -> 再想办法解释它和旧节点有什么区别`

### 2. 概念和模型怎么区分

概念回答：

- `这里到底出现了什么现象？`

模型回答：

- `这些现象背后反复出现的机制是什么？`

快判断：

- 能用一句话定义某个稳定现象，优先做 `concept`
- 需要把多个概念串成因果结构，优先做 `model`
- 如果它暂时只能解释这一集，先不要建节点

### 3. 不同关联字段，用的不是同一套键

这是当前项目最容易填错的地方。

- `episode.concepts` 填 `concept.id`
- `episode.models` 填 `model.id`
- `episode.people` 填人物 `name`，不是人物文件名
- `episode.themes` 填主题 `name`，不是主题文件名
- `episode.relatedEpisodes` 填 `EPxxx`
- `episode.tags` 填自然语言关键词，关键词页主要靠这里驱动

也就是说：

- 概念 / 模型：节目里用英文 slug `id`
- 人物 / 主题：节目里用中文展示名 `name`
- 关键词：不在节目 JSON 单独挂 `keywords` 字段，主要通过 `tags` 进入关键词系统

### 4. 关键词关联怎么生效

关键词层和其他节点不一样。

当前构建逻辑是：

1. 先从每个节目的 `tags` 自动生成关键词目录
2. 再把 `content/people/*.json` 自动并入关键词目录，人物会继承 `id / englishName / aliases / episodes`
3. 最后把 `content/keywords/*.json` 里的人工条目与前两层结果按 `name / aliases` 合并

所以：

- 想让某期进入某个关键词专题，首先检查 `episode.tags`
- 想让某个人物稳定进入关键词系统，先维护 `content/people/*.json`
- 如果只是同义词、口语词、下位词，不新增新关键词页，放进主关键词的 `aliases`
- 只有确实需要一个稳定专题入口时，才新建 `content/keywords/*.json`

### 4.1 `aliases` 和 `relatedKeywords` 不是一回事

- `aliases`
  - 表示同一个关键词的别名、口语、旧 slug、常见误写
  - 前台仍然落到同一个关键词页
- `relatedKeywords`
  - 表示两个关键词应该互相链接，但不应被视为同一个词
  - 前台要在关键词页显式展示“相关关键词”

判断标准：

- 如果用户搜索后应该进入同一个页面，用 `aliases`
- 如果用户点进去后应该看到不同页面，但两者需要互相跳转，用 `relatedKeywords`

例如：

- `新能源` / `电车` -> `新能源汽车`
  - 用 `aliases`
- `保险` / `保险公司` / `香港保险`
  - 不是同一个词
  - 应该各自保留页面
  - 但必须互相写进 `relatedKeywords`

### 4.2 构建阶段的关键词联动机制

- 构建时必须保留人工写入的 `relatedKeywords`
- 人工写入的 `relatedKeywords` 默认按双向关系处理
  - 例如在 `新加坡` 里写了 `黄循财`
  - 构建后 `黄循财` 也应自动反向出现 `新加坡`
- 构建时还要自动补一层“相近词联动”，而且这是主机制，不是附属点缀
- 自动联动至少覆盖三类：
  - 名称或别名互相包含
  - 明显的上位词 / 下位词
  - 同期节目高重合 / 高共现的关键词
- 已有 `parents` 关系也必须推出上下位连接
- 自动联动是主路径，人工 `relatedKeywords` 只负责：
  - 补足机器不容易看出来的关系
  - 修正高价值但低文本相似度的关系
- 一旦发现自动联动不够，必须回到 `content/keywords/*.json` 手工补 `relatedKeywords`

### 4.3 保险类词是第一组硬样例

以下三类词必须彼此联动，作为关键词联动机制的基准样例：

- `保险`
- `保险公司`
- `香港保险`

要求：

- 三者必须各自保留独立关键词页
- 三者必须互相出现在 `相关关键词`
- 后续新增类似词，例如 `航运保险`、`战争险`、`再保险` 时，也必须按同样规则处理：
  - 先判断是否独立成页
  - 如果独立，就补进相关词网络

### 5. `episodes[].note` 怎么写

无论是概念、模型、人物、主题还是关键词里的 `episodes` 数组，`note` 都不要写成摘要复读。

正确写法：

- 说明这期为什么和这个节点有关
- 说明它承接的是哪一个判断或哪一种机制

错误写法：

- `本期讨论了这个概念`
- `这期和这个模型有关`

正确方向：

- `节目把维文的表态视作小国守住规则边界的典型表达`
- `节目把保险、护航和定价权合并解释为海峡秩序的一部分`

### 6. 模型的 `sourceLabel / sourcePath` 规则

模型条目必须明确自己是本地颖响力模型，还是复用 GV 现有模型。

如果是本地颖响力模型：

- `sourceLabel`: `颖响力模型：模型中文名`
- `sourcePath`: 当前模型 JSON 的绝对路径

如果是复用 GV 模型：

- `sourceLabel`: `GV 模型：模型中文名`
- `sourcePath`: 指向 GV 原始模型卡路径

不要空着这两个字段，也不要随便写一句说明文字代替。

### 8. 从节目到节点的回填顺序

标准顺序固定为：

1. 先把 `content/episodes/EPxxx.json` 写完整
2. 再决定哪些内容值得沉淀成 `concepts / models / people / themes / keywords`
3. 新建或复用节点后，先把该节点自己的 `episodes` 和 `related*` 补齐
4. 再回填节目里的：
   - `concepts`
   - `models`
   - `people`
   - `themes`
   - `relatedEpisodes`
5. 最后跑 `npm run build` 验证网页跳转

不要先建一堆节点，再倒推节目内容。

### 9. 新节目如何接入旧知识库

这一步必须显式做，不能只把新节目写完就停。

每来一集新节目，都要回答同一个问题：

`这期内容应该挂回旧库里的哪些节点？哪些词要复用，哪些词要新建，哪些词之间需要补链接？`

标准流程固定为：

1. 先写新节目的核心结构
   - 完成 `content/episodes/EPxxx.json` 的：
     - `summary`
     - `topic`
     - `viewpoints`
     - `extensions`
   - 先把这一期真正讲了什么说清楚，再决定知识库链接

2. 再做“旧节点搜索”，不要凭记忆
   - 至少查：
     - `content/concepts`
     - `content/models`
     - `content/people`
     - `content/themes`
     - `content/keywords`
   - 推荐先用：

```bash
rg -n "关键词A|关键词B|关键词C" content/concepts content/models content/people content/themes content/keywords
```

3. 对每一个新出现的重要词，强制做三选一判断
   - `并入 aliases`
     - 如果它只是旧关键词的别名、旧写法、口语词、戏称
   - `并入 relatedKeywords`
     - 如果它和旧关键词不是同一个词，但应该互相跳转
     - 典型情况：
       - 上位词 / 下位词
       - 同一议题簇里的近邻词
       - 人物 / 国家 / 公司 / 产品之间的稳定专题关系
   - `新建独立节点`
     - 只有当它已经足够稳定，能承接未来多期节目时才新建

4. 先补节目到旧库的正向链接
   - 回填新节目的：
     - `concepts`
     - `models`
     - `people`
     - `themes`
     - `tags`
     - `relatedEpisodes`
   - 这一层决定新节目怎样进入旧知识网

5. 再补旧节点到新节目的反向链接
   - 对被命中的概念 / 模型 / 人物 / 主题 / 关键词：
     - 把 `EPxxx` 写进对应节点的 `episodes`
     - `note` 说明这期为什么和它有关
   - 不允许只有节目单向引用旧节点，而旧节点不记这期

6. 最后补词与词之间的联动
   - 如果新节目带来了新的词簇关系，必须补到 `content/keywords/*.json`
   - 优先补：
     - `aliases`
     - `relatedKeywords`
     - 必要时 `parents`

7. 最后验证“点进去之后能不能顺着走”
   - 至少自己检查：
     - 从新节目点进关键词
     - 从关键词点回相关节目
     - 从关键词点到相关关键词
     - 从人物/国家/公司这种词簇里能不能继续横向跳

#### 9.1 一个新节目接入旧库时，最少要补哪几种链接

每期至少检查这五类：

1. 这期对应哪些旧节目
   - 填 `relatedEpisodes`

2. 这期对应哪些旧概念 / 旧模型
   - 复用旧 `concepts / models`

3. 这期对应哪些旧人物 / 旧主题
   - 回填 `people / themes`

4. 这期带来的关键词，应该并入哪个旧关键词
   - 能并入 `aliases` 就不要新开页

5. 这期新增的关键词关系，是否需要补 `relatedKeywords`
   - 只要两个词不是同一页、但用户应该能互相跳，就补

#### 9.2 `relatedKeywords` 的使用标准

不要把 `relatedKeywords` 当装饰字段。

它的职责是：

- 把本来分散的近邻词重新织回知识网
- 避免用户搜到一个词后，永远停在单页里

必须优先补 `relatedKeywords` 的情况：

- 上位词 / 下位词
  - 例如：
    - `保险` ↔ `保险公司`
    - `新能源汽车` ↔ `小米汽车`
- 国家 / 核心人物 / 关键制度
  - 例如：
    - `新加坡` ↔ `黄循财 / 李显龙 / 维文 / 组屋`
- 议题主词 / 关键载体
  - 例如：
    - `伊朗` ↔ `霍尔木兹海峡 / 哈梅内伊`

#### 9.3 新节目接入后，必须做的最终检查

每次新节目接入后，至少做一次：

```bash
npm run build
```

然后人工检查：

- 新节目的详情页是否已经挂回旧库
- 被命中的关键词页是否已经出现这期
- 相关关键词是否已经能互相跳转
- 是否还有明显孤立词没有被并回旧知识网

## 关键词整理规则

关键词层的目标不是“把字幕里所有名词都做成平级入口”，而是优先形成稳定、好用的专题入口。

### 1. 优先保留上位专题，不滥开平级同义入口

- 优先保留真正能承接节目群的专题词，例如：
  - `娃哈哈`
  - `新能源汽车`
  - `西贝`
- 不要把明显属于同一簇的近义词、口语词、下位表达全部做成平级专题。

例如：

- `新能源 / 新能源汽车 / 电车`
  应优先收束到 `新能源汽车`
- `宗馥莉 / 宗庆后 / 娃哈哈`
  如果当前讨论主线是娃哈哈家族与公司问题，应优先收束到 `娃哈哈`
- `杂粮汽车 / 小米汽车`
  应收束到正式入口 `小米汽车`

### 2. 别名和口语词放到 `aliases`

- 如果一个词只是别称、口语、戏称、缩写，不要再单独做一个平级关键词页
- 把它放进主关键词的 `aliases`

例如：

- `新能源`、`电车` 作为 `新能源汽车` 的 `aliases`
- `哇哈哈`、`宗馥莉`、`宗庆后` 作为 `娃哈哈` 的 `aliases`
- `杂粮汽车` 作为 `小米汽车` 的 `aliases`

### 3. 人物默认并入关键词层，不再做独立人物页

- `content/people` 负责存人物源数据
- 前台人物入口统一落到 `#/keywords/...`
- 导航里的 `人物` 只是关键词层里的一个自动分组，不是独立数据源
- 搜索里的 `人物` 结果也默认落到对应人物关键词页；只有找不到对应关键词时，才允许退回 `people/...`
- 自动规则：只要关键词被识别为人物，且被引用至少 `2` 次，就应自动进入 `#/people` 和侧边栏人物导航
- 如果人物只是服务于某个公司、品牌、家族或事件主线，优先并入上位关键词，不单独占关键词入口
- 如果人物本身在多期节目里已经形成独立讨论线，可以保留独立人物关键词，但仍然通过关键词页访问

判断标准：

- 问题核心是不是“这个人本身”
- 还是“这个人只是某个专题里的关键角色”

### 4. 上位词与下位词不要乱并列

- `汽车` 可以是上位词
- `新能源汽车` 是更具体的行业/产品专题
- `电车` 是更口语化、消费语境更强的表达

整理时优先选择最能代表该节目主线的那一个，不要同时堆多个高度重叠的泛词。

### 5. 单次出现的长尾词先不在关键词索引页展示

- 只出现 `1` 次的词可以保留在数据层，便于未来累计
- 但 `#/keywords` 列表页默认不展示
- 关键词索引页优先展示至少出现 `2` 次、已经形成节目群的词
- 人物导航也沿用这个门槛：至少出现 `2` 次的人物关键词才进入导航分组

### 6. 自动生成关键词文案不要写废话

- 不要写：
  - `XX相关关键词，聚合知识库里提到这个主题的N期节目`
- 应该写成一句真正有信息量的专题介绍

例如：

- `围绕房价的节目群入口，用来串联地产周期、政策压力和市场预期的相关讨论。`
- `围绕新加坡的节目群入口，用来串联其地缘位置、治理模式和区域博弈中的角色。`

### 7. 新整理节目时，先问自己这几个问题

1. 这个标签是不是已经属于现有专题？
2. 它是主入口，还是别名？
3. 它是独立专题，还是上位专题里的子人物 / 子问题？
4. 它现在只出现一次，是否值得先留在数据层、暂不放进列表页？

## 网页交互规范

### 返回按钮

- 详情页保留 `返回前一页`
- 同级索引返回按钮保留：
  - `返回概念页`
  - `返回模型页`
  - `返回主题页`
  - `返回关键词页`
- `返回前一页` 可以保留箭头
- `返回概念页` 这类按钮不带箭头，文案更干净
- `返回首页` 与 `返回概念页` 这类按钮使用同样的浅橙色样式

### hover 反馈

- 返回按钮需要有轻微 hover 变色
- 折叠项标题需要有轻微 hover 变色
- 可点击卡片、列表项、关联卡片都需要有轻微 hover 变色
- 原则是“让人感知得到可点击”，但不要做成厚重按钮感
- 如果 hover 后背景进入深色方案，标题、正文、标签都必须同步切到浅色高对比文本
- 禁止出现“深色背景 + 棕色/金色标题仍然偏暗”的搭配，深底可读性优先于品牌色强调
- 知识页头部与正文里的引用 chip，默认允许保持浅底，但 `hover / focus` 必须切到深色底，明确表达“这是可点击入口”
- 禁止出现“默认态和 hover 态几乎一样，用户分不出能不能点”的知识 chip

### 索引页导语

- 概念 / 思想模型 / 人物 / 主题索引页顶部说明必须服务于“怎么使用这个入口”
- 禁止写实现细节、筛选条件复述或整理者视角的话，例如：
  - `这里只显示被引用超过 1 次的概念`
  - `当前规则是自动从关键词里识别人物`
  - `按引用率排序 / 每类先显示 3 个`
- 导语应直接回答读者一个问题：
  - 我为什么从这里进入？
  - 这个入口和节目页相比，多提供了什么阅读路径？
- 概念 / 模型 / 人物 / 主题索引页默认不自动展开第一组，优先让用户自己选择要先看哪一类

### 左侧导航规则

- 左侧导航以“查找优先”为目标，不堆太长列表
- `关键词` 默认只显示 `3` 个
- `核心概念` 默认只显示 `5` 个
- `思想模型` 默认只显示 `5` 个
- `更多模型` 下方单独新增一个 `知识网` 分组
  - 分组标题写 `知识网`
  - 分组内保留 `知识图谱` 入口
- 每个分组下方保留 `更多` 入口，跳转完整索引页
- `已整理节目` 不再放在左侧导航中

### 关键词搜索规则

- 左侧保留 `搜索关键词` 输入框
- 输入框下方默认显示 `推荐关键词`
- 推荐关键词默认不超过 `3` 个
- `推荐关键词` 不能固定写死
  - 必须带随机性
  - 但随机权重优先给：
    - 引用次数更高的关键词
    - 关联最新节目的关键词
- 开始输入后，标题切换为 `匹配结果`
- 左侧搜索不只搜关键词，也搜节目
- 输入 `31`、`31集`、`EP31`、`三十一集` 时，应能直接找到 `EP031`
- 按回车时优先跳到第一个精确命中的节目或关键词页

### 索引页排序与折叠规则

- `#/episodes` 的节目列表与相关节目列表默认按集数从大到小排序
  - 最新节目放在最前面
- `#/episodes` 的集数工具条必须同时包含：
  - 集数区间导航
  - 节目搜索
- 集数工具条桌面端和手机端必须分开设计，不能只把同一套布局硬缩小
  - 桌面端可以保留压缩式分页
  - 手机端必须使用更小的极简版本，把空间优先留给节目内容本身
- `#/episodes` 的默认态固定为：
  - 集数区间导航在左
  - 搜索入口缩成一个放大镜按钮，固定在右
  - 默认态不允许长期把完整搜索框顶在工具条里占位
- 首页右侧悬浮快捷操作固定为四个按钮：
  - 导航
  - 节目搜索放大镜
  - 首页
  - 返回顶部
- 首页右侧悬浮快捷操作里的节目搜索放大镜必须与 `#/episodes` 右侧放大镜使用同一条进入逻辑
  - 无论当前在哪个页面，点首页悬浮放大镜后，都必须直接进入节目索引的搜索态
  - 进入后的状态必须等同于“用户在 `#/episodes` 页里点右侧放大镜”
  - 不能只把用户送到 `#/episodes` 列表页而不展开搜索
- 首页右侧四个悬浮按钮的间距要优先避免误触
  - 如果新增按钮后空间变紧，优先加大按钮间距与点按间隔，而不是优先压缩成挤在一起
- `#/episodes` 的默认态里，集数区间导航和放大镜必须稳定放在同一行
  - 放大镜必须单独占一个固定宽度列
  - 右侧必须提前留出足够空白，不要把最后一个完整区间标签挤成半截
  - 调整空间时，优先控制可见区间数量，不要让 `81-90集` 这种完整标签被切半
- `#/episodes` 的放大镜图标固定为右把手的放大镜
  - 默认态要明显、够粗
  - 不能用方向含混的字符图标糊过去
- 集数工具条必须尽量薄，不能遮住下方搜索结果或节目列表主体
- `#/episodes` 的搜索输入不能在打字时把页面跳到别的段落
  - 搜索更新必须尽量局部化
  - 必须保持当前滚动位置和输入光标位置稳定
- `#/episodes` 的搜索态固定为：
  - 点放大镜后再展开搜索框
  - 搜索框展开时，可以临时收起默认态标题区或区间条，但退出后必须恢复
  - 搜索框必须保持打开，直到用户主动退出；不允许因为定时器自动收起
  - 点击工具条外空白时，应退出搜索态
- `#/episodes` 的搜索必须按“覆盖层”处理，而不是按“重进一个新页面”处理
  - 如果用户从别的页面调起节目搜索，但没有真正进入节目详情或明确结果，退出搜索后必须回到原来的页面
  - 回到原页面时，必须尽量恢复原来的滚动位置，不允许先跳到顶部再退回
  - 如果用户本来就在 `#/episodes` 里打开搜索，退出后必须回到刚才看的节目索引位置，不要重置到列表顶部
- `#/episodes` 的搜索流程固定为：
  - 搜索框空态先显示 `推荐关键词`，右侧配 `换一换`
  - `推荐关键词` 默认只显示一小组，不要无限下探；优先短而准
  - 输入后，上方建议区只显示知识条目建议：关键词 / 人物 / 主题 / 概念 / 模型
  - 上方建议区不再重复混入节目建议；节目结果统一留在下方节目列表区
  - 输入时，下方节目列表就应直接同步显示匹配到的节目，不必等用户先点建议词
  - 推荐关键词点击后，应把词带进搜索框，并在当前搜索态里展示相关节目；不能直接跳知识页
  - 上方知识建议点击后，再跳对应知识页
  - 建议项右侧的类型标记必须固定贴右，不能和左侧词条文本粘在一起
  - 如果同一搜索结果里 `人物` 和 `关键词` 最终指向同一个词条，只保留一条，优先显示 `人物`
- `#/episodes` 的节目轮盘与首页章节轮盘不同：
  - 在节目索引页里点击轮盘条目后，必须直接进入对应那一集的详情页
  - 不能只把页面滚到当前列表中的某一段
- `#/episodes` 底部左右切组按钮必须比普通胶囊按钮更大
  - 手机端要优先保证点按面积，而不是优先追求纤细
- `#/episodes` 顶部标题区必须同时保留：
  - `返回前一页`
  - `返回首页`
  - 不能只给 `返回首页`，否则用户从别的页面进入节目索引后没有自然返回路径
- 集数工具条的悬浮透明交互固定为：
  - 刚到原始位置时，正常显示
  - 向下滑过原始位置后，按滚动距离逐渐变透明
  - 鼠标 `hover` 到透明工具条时，直接恢复全亮
  - 点击或聚焦工具条时，也直接恢复全亮
  - 鼠标移开且继续下滑时，再继续按滚动位置变透明
  - 向上滑回原始位置后，恢复正常颜色
- `#/keywords` 的关键词专题卡片底部信息要固定在卡片底边，不允许高低错位
- 首页 `节目索引` 卡片底部的小标签也必须固定在卡片底边，不允许因为摘要长短不同而上下漂移
- 节目标题展示必须自动清洗尾部重复编号
  - 例如标题末尾的 `【EP117】`
  - 这类重复编号不能再出现在已经带 `EP117｜` 前缀的标题展示里
- `#/keywords` 只显示至少出现 `2` 次的关键词
- 单引用关键词虽然默认不在 `#/keywords` 列表页显示，但也必须写成可独立成立的入口说明
  - 不能再使用“从节目标签自动沉淀出来的关键词入口”这类自动废话
  - 单引用关键词至少要写清：
    - 它当前锚定的是哪一期节目
    - 它为什么值得保留成一个独立关键词入口
- 如果单引用关键词本身是概念性词、价值判断词、文明/归属类词，例如：
  - `认祖归宗`
  - `小国平衡`
  - `故乡`
  这类词不能继续依赖自动生成文案，必须转成人工关键词文件，真正解释这个词本身在知识库语境里是什么意思
- 关键词页写法要按“科普词条”来写，不要写成一整段难以阅读的长坨文字
  - `summary`：一句话，先回答“这个词是什么”
  - `description`：分段写
    1. 第一段解释这个词本身
    2. 第二段再说它在颖响力语境里为什么重要
  - 每一段尽量短，不要一段塞太多转折和并列信息

### 8. 关键词页必须是“讲给读者听”的，不是后台备注

这一条是硬规则。

关键词页文案必须像正常词条，直接讲给网页读者听，不要写成整理者对自己说话。

禁止出现这类后台口吻：

- `先把它保留下来`
- `后续可以继续往这个入口里累积`
- `当前锚定`
- `最直接锚定`
- `这个词更多指向`
- `值得保留`
- `这里不只是一个标签`
- 任何像“我为什么建这个词条”的说明

用户在看词条时，只关心两件事：

1. 这个词到底是什么
2. 这档节目里为什么会讲到它

所以正确写法应该是：

- 先用正常人话把词解释清楚
- 再直接写 `在 EPxxx 这期里，节目把它放进……` 或 `在这期讨论里，它主要用来说明……`
- 只写节目里的实际判断，不写整理流程、不写建库理由、不写后续维护计划

### 9. 关键词页不能字太少，必须有最小信息量

很多词条的问题不是方向错，而是字太少，读者看完还是不知道它到底在讲什么。

最低要求：

- `summary` 不能只有十来个字的标签式解释
- `summary` 至少要做到：
  - 说清这个词是什么类型的东西
  - 说清它在节目语境里大致落在哪个问题域
- `description` 必须至少两段
  - 第一段解释词本身
  - 第二段解释节目里为什么会讲到它
- 第二段不能只写：
  - `在 EPxxx 这期里提到过`
  - `节目讨论了这个问题`
  - `它很重要`

第二段必须补出一条具体判断，例如：

- 节目到底拿它解释了什么现象
- 它在那期里是风险开关、权力结构、政策工具，还是某种叙事包装
- 为什么它不是随手一提，而是那期判断链条里的关键部件

简单说：

- 不是把词认出来就算完成
- 而是要让用户点进去以后，真的“看懂这个词”

### 10. 关键词整改必须跑审计，不要停在“先改一批”

关键词页整改不是修几个明显错词就结束。

每次批量改完后都必须执行：

```bash
npm run build
npm run audit:keywords
```

处理顺序固定为：

1. 先改最像后台备注、最不像人话的词条
2. 再改字太少、读者看完仍然不知道在讲什么的词条
3. 再改剩下那些虽然不算错、但信息密度不够的词条

默认要求：

- 不要停在“已经比以前好一点”
- 要持续按审计清单往下清
- 每清一批就重新 build + audit
- 看新的剩余数量，再继续下一批

### 11. `scripts/keyword-definitions.d/*.json` 是关键词页的人类文案覆盖层

关键词页的人话文案，统一落在：

- `scripts/keyword-definitions.json`
- `scripts/keyword-definitions.d/*.json`

其中分批整改时，优先往 `keyword-definitions.d` 里追加新 batch 文件。

这一层的职责是：

- 覆盖自动生成的关键词文案
- 覆盖人物/概念/主题继承过来的过短摘要
- 把页面改成真正能给读者看的词条

不要以为：

- `content/people`
- `content/concepts`
- `content/themes`

里已经有一句摘要，就算关键词页写完了。

关键词页最终展示效果，以这层人工定义为准。
- `#/concepts`、`#/models`、`#/people`、`#/themes`
  - 只显示被引用超过 `1` 次的条目
  - 默认按引用率从高到低排序
  - 按类别分组
  - 每个类别先显示前 `3` 个
  - 其余条目折叠到 `更多`
- `#/people`
  - 人物分类优先使用人工映射，其次才允许规则兜底
  - 像 `高市早苗`、`特朗普` 这类边界人物，不能只靠关键词猜分类
- `#/models`
  - 分类要比其他页更细，优先按机制层区分，而不是只分几个大筐
- 单引用条目虽然默认不在索引页显示，但仍然必须按对应类型的 SOP 写完整
  - “不显示” 不等于 “可以只留一个壳”
  - 隐藏条目也必须具备独立可读性，后续一旦引用数增长可以直接露出

### 固定图标与入口规则

- 全站右下角固定保留两枚按钮：
  - 回首页
  - 返回顶部
- 两个按钮桌面端和手机端都固定在右下角
- 返回顶部按钮默认常驻显示，不需要滚动后才出现
- 首页标题下面固定显示平台主页入口：
  - `bilibili`
  - `YouTube`
- 平台主页入口放在标题正下方，不放在标题右边
- 手机端左上角三明治按钮默认使用更浅、更透明的样式，减少对正文遮挡
- 侧边栏打开时，三明治按钮再恢复为深色高对比状态
- 节目页标题下方的视频入口也是固定规范：
  - 普通 B 站：蓝色入口
  - 会员 B 站：金色入口
  - 已下架 B 站：灰色入口，hover 显示 `已下架`
  - YouTube：固定为 YouTube 风格入口
  - 平台入口必须有 hover 反馈
- 节目页顺序固定为：
    1. 标题
    2. 摘要
    3. 标签
    4. 视频平台入口

### 字体与职业感规则

- 整站字体要优先体现知识库/文档站的职业感，而不是泛博客感或系统默认感
- 正文优先使用稳定、专业的中文无衬线栈
  - 优先 `Noto Sans CJK SC / Source Han Sans SC`
  - 再回退到 `PingFang SC / Hiragino Sans GB / Microsoft YaHei`
- 标题优先使用更克制的中文衬线栈
  - 优先 `Noto Serif CJK SC / Source Han Serif SC`
  - 再回退到 `Songti SC / STSong`
- 正文和标题不能继续沿用偏个人博客感的随意混搭英文字体作为主字体方案
- 标题、正文、说明、标签必须是四层不同语气，不能做成“所有字像一段话”
  - 页面主标题：最大、最紧、最有识别度
  - 模块标题 / 卡片标题：次一级，但仍然明显区别于正文
  - 正文 / 摘要：稳定、退后、可长时间阅读
  - 标签 / 说明 / eyebrow：最小、最轻、最克制
- 标题和正文不只靠字号区分，还要同时拉开：
  - 字体家族
  - 字重
  - 行高
  - 字色
  - 字距
- 卡片标题、列表标题在移动端必须一眼看出是“标题”，不能和下面摘要融成一个段落
- 正文颜色应比标题更退后，避免标题和正文同黑同重
- 说明类文字、section note、inline label、eyebrow 必须统一走更小、更浅、更高字距的辅助层，不要和正文抢注意力

### 滚动与对齐规则

- 网页端和手机端都优先保证自然、轻量的滚动手感，不做重度 scrolljacking
- 手机端长页面滚动应尽量自然落在大块内容起点附近，避免停在半张卡片中间
- 首页 hero、首页 section、详情页 header、详情页 section 应使用一致的轻量滚动对齐策略
- 如果使用滚动对齐，优先使用 proximity，而不是强制 snap
- 每次改动滚动体验，都必须本地验证桌面端和手机端截图，不只看代码
- 自动辅助滚动只在用户已经接近模块边界时轻量介入，目标是把页面送到最接近的模块起点，而不是强拉到固定位置
- 如果上下两个模块同时卡在屏幕中间、两边都露出半截内容，应自动选择更合理的落点，避免页面停在“两边都不完整”的中间态
- 自动辅助滚动必须结合当前滚动方向、可见面积和距离一起判断，不能只靠单一阈值
- 文档底部或用户明显还在主动拖动时，禁止反向拉回
- 快速滚动时允许出现极轻的短时模糊和章节轮盘提示，但轻微滚动不应明显模糊
- 模糊的退场必须早于轮盘退场，不能让页面在用户已经停下后还长时间发糊
- 悬浮搜索栏在用户上滑时可以回显，但如果用户没有继续交互，应在短暂停留后自动收起，避免长期遮挡正文
- 知识详情页（概念 / 模型 / 主题 / 关键词 / 人物）首屏不应再跑额外 reveal 入场动画
  - 页面路由完成后，标题区与首屏正文必须一次性稳定落位
  - 禁止出现“页面已经到了，但标题区又闪一下 / 跳一下”的二段动画感
- 知识详情页之间仍然应保留极轻的切换过渡
  - 目标是“有一点网页切换感”，不是“完全没动画”
  - 只允许短时淡入淡出与极小位移
  - 禁止恢复重 blur、重缩放、长距离位移、翻页感过强的动画

### 首页交互边界

- 首页 `节目索引` 的左右按钮、滑动手势、自动轮播必须视为同一套状态机
  - 三者都只能驱动同一个 `currentIndex`
  - 每次只允许移动 `1` 个节目
  - 自动轮播等价于点一次“下一张”
- 首页节目索引的响应式判断必须按内容区真实宽度决定 `3 / 2 / 1`
  - 宽时 `3` 张
  - 中间宽度 `2` 张
  - 更窄 `1` 张
  - 任何时候都不允许出现两排
- 首页节目索引的首屏页数判断必须稳定
  - 首次渲染与完成布局后的最终渲染必须使用同一套宽度基准
  - 不允许出现“首屏先显示 3 张，随后又掉成 2 张”这类跳变
  - 需要优先排查滚动条 gutter、容器真实宽度、首屏 fallback 宽度不一致这三类原因
  - 如果边界宽度容易抖动，应优先稳定宽度来源，而不是让首页反复重判
- 首页节目索引切换时禁止整页重渲染
  - 只能局部更新轮播区域
  - 否则容易把搜索栏、滚动位置、焦点状态一起打乱
- 首页节目卡片高度必须收敛到稳定区间
  - 标题和摘要允许裁切，但卡片外框高度要稳定
  - 不能因为某一期标题更长、摘要更满，就把 `概念入口` 和后续区块整体往下顶
- 首页节目卡片必须按“标题 / 简介 / 关键词”三段式稳定分配高度
  - 标题允许比普通卡片更长，但必须有固定最大行数；超出时用 `...`
  - 简介也必须有固定最大行数；内容短时完整显示，超出时再截断
  - 关键词区必须固定在卡片底部，不能被长标题或长简介挤出边框
  - 如果某一期标题特别长，优先裁掉标题和简介的尾部，不要牺牲关键词区可见性
  - 首页卡片默认只展示可控数量的关键词，不要为了“全展示”把底部区域撑坏
- 新节目状态必须自动计算，不靠手工改字
  - 以节目原始源文件时间为准
  - 最近 `3` 天内：
    - 首页节目卡片把 `已整理` 替换成红色 `新`
    - 节目索引卡片里的节目编号旁边也显示红色 `新`
    - 内容页标题后面显示一个小号红色 `新`
  - 超过 `3` 天后：
    - 首页自动恢复成 `已整理`
    - 节目索引卡片里的 `新` 自动消失
    - 内容页标题后的 `新` 自动消失
  - 不允许依赖“过几天手工再改一次”
- 首页节目卡片的状态文案只允许使用：
  - `新`
  - `已整理`
  - `待整理`
- 会员状态不允许替代首页节目卡片的整理状态
  - 会员信息只在视频入口层表达
  - 首页卡片不要显示 `会员` 去覆盖 `已整理` 或 `新`
- 但“稳定”不能以牺牲主标题可读性为代价
  - 手机端节目标题优先保证完整显示
  - 如果必须压缩空间，先降字号、调行高、减少摘要行数，而不是先裁掉标题
- 自动轮播、按钮切换、手势切换都不得改写当前滚动位置
  - 不能把页面突然拉回搜索区
  - 不能触发章节吸附把页面带偏
- 手机端节目索引不允许只做“松手后换页”的硬切
  - 必须支持真实左右拖动
  - 拖动过程中要能看到前一张 / 后一张的边缘预览
  - 视觉上更接近翻页，而不是整块闪切
- 但如果手机端节目索引的主卡被左右按钮和边缘预览压得过小，应优先重做交互模式，而不是硬保留桌面版结构
  - 手机端允许取消左右箭头
  - 手机端允许取消两侧边缘预览
  - 目标是保证主卡正文面积、标题可读性和点击成功率
- 手机端首页节目索引优先采用“单卡全宽 + 左右滑动切换”的工作模式
  - 主卡必须尽量吃满可用宽度
  - 左右滑动负责切换节目
  - 点按负责跳转，不允许和滑动职责混在一起
- 手机端首页节目索引的滑动手感必须连续
  - 达到翻页阈值后，不允许先往回弹一下再翻过去
  - 视觉上要像顺着手势继续完成翻页，而不是“回抽一下再补动画”
  - 如果手感发涩，优先调横向识别阈值、位移预览幅度和翻页触发阈值
- 手机端首页节目索引必须提供明确、稳定的打开入口
  - 点整张卡可以打开
  - 点标题可以打开
  - 如果交互上仍可能误触，允许额外提供单独的 `打开节目` 按钮
  - 不允许出现“点了好几次都没反应”
- 手机端首页节目索引的自动轮播可以保留，但频率要更慢
  - 默认按 `10 秒` 一次处理
  - 不能频繁打断阅读
- 用户正在拖动或点节目索引按钮时，自动轮播必须暂停
  - 不能一边用户在操作，一边系统自己换页
  - 自动轮播只在用户没有继续干预时恢复
- 手机端首页节目索引的摘要不能继续直接塞原始长摘要
  - 需要单独生成一版移动端摘要
  - 目标长度接近原摘要的 `75%`
  - 优先按句子边界或停顿边界收尾
  - 禁止出现“半句被硬截断悬在那里”
  - 但也禁止压缩成过短口号式文案
- 首页搜索输入时，搜索栏和结果区必须一起进入可见区
  - 不能只把输入框顶到上面而让结果区留在原位置
  - 搜索栏与结果区之间保留层次，不做成一整块黏连
- 首页搜索栏的显隐边界固定为：
  - 回到搜索区原始位置时稳定可见
  - 离开搜索区后，上滑可短暂回显
  - 停住不继续交互时自动收起
  - 节目轮播切换不得影响它的显隐状态
- 右侧章节轮盘不应在首页首屏直接出现
  - 进入后续内容区后再出现
  - 轮盘提示退场应晚于轻微滚动停止，但不能长期停在屏幕上
- 右侧三按钮展开后允许短暂保留，但无操作时必须自动收回
  - 收回后只保留最小入口
  - 不得打断正在点击的按钮
- 左侧导航最终结构固定为：
  - 主导航：`首页 / 节目 / 概念 / 模型 / 人物 / 主题 / 关键词`
  - 搜索区只保留 `搜索知识库`
  - 底部单独保留 `知识图谱`
  - `网页日志` 固定在 `知识图谱` 下方
  - 不再恢复旧的重复折叠块结构
- 首页不使用过度保守的延迟显现策略
  - 首页模块在初次渲染后应尽快全部可见
  - 不允许用户已经滚到页面中段，`概念入口 / 思想模型 / 知识图谱` 仍然大面积留白
- 首页视觉层级不允许长期停留在“整页同一层浅色”
  - 至少通过分区底色、边框强弱、文字层级三种手段把不同模块分开
  - 默认态就要能分出主区块，不要把“只有 hover 才有层次”当成常态
  - 但分区底色必须保持低饱和、统一气质，禁止做成花花绿绿的廉价拼色
  - 分区之间只能做轻微温度差和明度差，不要做强烈彩色块
- 知识详情页里的 `相关节目` 证据层必须明确区分“节目卡本体”和“关键词入口”
  - 桌面端默认态：节目卡浅底深字，关键词 chip 深底浅字
  - 桌面端 `hover / focus`：节目卡切深底浅字，关键词 chip 反转成浅底深字
  - 移动端可按空间与可读性反转主次，但仍必须保持“节目卡”和“关键词 chip”一眼可区分
  - 节目标题默认不使用多余下划线装饰，优先靠颜色、层级和 hover 反馈表达主入口
  - 不允许出现“节目卡和关键词都是同一明暗逻辑，导致主入口和次入口混在一起”的状态
  - 如果某条规则导致浅底浅字、深底深字或 hover 后反差错误，必须以可读性为准改掉旧规则
- 正文里的 `EPxxx` 内联说明浮窗必须是全局浮层，不得受 accordion、卡片或 section 容器裁切
  - 浮窗优先保证留在视口内
  - 右下放不下就换右上，再不行换左侧，不允许固定朝一个方向溢出
  - 浮窗逻辑服务正文分析区，不要反过来干扰证据层节目卡
- 知识详情页右侧章节轮盘在 `相关节目` 区内，必须支持 EP 级别分项
  - 不能只把整块 `相关节目` 当成一个笼统 section
  - 至少要能看到 `EP122`、`EP090` 这类条目并支持直达

### 首页移动端节目索引硬规则

- 手机端首页 `节目索引` 的主卡优先级高于边缘预览、箭头装饰和花哨轮播感

  1. 左右箭头
  2. 边缘预览
  3. 多余装饰
  4. 最后才考虑压缩标题和摘要

- 手机端首页节目索引摘要允许比桌面短，但必须像正常摘要，而不是像残句
- 手机端首页节目索引点按测试必须单独验证：
  - 轻点整卡能打开
  - 轻点标题能打开
  - 轻点按钮能打开
  - 左右滑动切换时不会误开
- 首页头像必须完整显示
  - 不允许出现“头像只露出顶部一条”或被默认裁切规则截掉的状态
  - 如果头像源图不是刚好适配圆形容器，优先调整显示方式让头像完整可见，而不是强行 `cover` 裁掉脸部

## 原始源文件与标题标准

### 标题命名标准

- 原始节目源文件统一使用：
  - `[EPxxx】 正式标题.srt`
- 标题主体只保留节目编号和正式标题，不要再把：
  - `【EP123】`
  - `｜关键词尾巴`
  - `#话题标签`
  重复塞回标题末尾
- 如果你手里只有聊天整理稿、手打文字稿、AI 转写整理稿，没有时间轴，先把它整理成顺排的 `.srt`
- 这套知识库的标准原始字幕文件仍然是 `.srt`
- 节目标题里如果出现 `｜` 或 `|`，后面的部分一律视为标签尾巴，不算标题正文
- 页面展示标题时，默认只显示 `｜` 前面的标题正文
- `｜` 后面的尾巴应优先进入：
  - `tags`
  - 关键词入口
  - 平台视频标题匹配辅助

例如：

- `[EP123】 伊朗为何是现代版“三国荆州”？诸葛亮庞统之争看懂大国博弈底层逻辑.srt`

### 只有文字稿时，怎么转成可导入的 `.srt`

- 如果已经有标准字幕时间轴，直接保留原 `.srt`
- 如果只有纯文字内容、分段整理稿、聊天里发来的讲稿，就先转成顺排 `.srt`
- 对于知识库导入来说，时间轴精确到真实口播不是第一优先级，关键是：
  - 文件名合规
  - 文本内容完整
  - 时间轴格式合法且顺序递增
- 导入进入知识库后，脚本会去掉时间轴，抽成纯文本 transcript 进入 `workbench/EPxxx/`

## 节目内容怎么做

这一部分是整个知识库最重要的工作面。不要把节目条目理解成“字幕摘要”，而要把它理解成：

- 这一期到底在解释什么
- 它的主判断是什么
- 它把哪些现象连成了一条因果链
- 哪些节点值得被沉淀成长期可复用的概念、模型、人物、主题

### 基本原则

- 先抓“节目真正想说什么”，再写字段
- 不要按字幕顺序机械抄录
- 不要把一堆正确废话拆成 bullets
- 不要写成整理者备注
- 要写成“用户点开就能读懂这一期在讲什么”

### 节目页的黄金标准

当前节目条目的参考标准是：

- `EP001`

它的特点不是字多，而是结构完整：

- 有明确主判断
- `topic` 不是空架子
- `viewpoints` 每条都有独立判断价值
- `concepts / models / people / themes / relatedEpisodes` 都不是装饰
- 节目本身能反向推动节点层变厚

整理新节目时，默认以 `EP001` 的完成度为目标，而不是以“能显示出来”为目标。

## 如何从 transcript 提炼节目主线

### 第一步：先找主判断

先回答一句话：

- 这期最想说的核心判断是什么？

如果这句答不出来，就不要急着填 JSON。

正确做法：

- 先把节目里的判断压缩成一句话
- 这句话必须带方向感，而不是中性复述

例如：

- 不要写：`节目讨论了银行直供房、法拍房和房价问题`
- 要写：`节目把银行直供房视为法拍通道装不下风险后，银行为回收流动性开辟的新处置路径，而不是楼市见底信号`

### 第二步：再找它的论证链

问自己：

1. 它拿什么现象开头
2. 它把这些现象解释成什么机制
3. 它最后落到什么判断或建议

通常可以整理成：

- 起点现象
- 核心矛盾
- 解释机制
- 最终落点

### 第三步：只保留节目真正强调的部分

不是 transcript 里出现过的每句话都值得进结构化条目。

优先保留：

- 反复出现的判断
- 串联多个细节的解释框架
- 节目最后明确落下来的结论
- 能沉淀为概念/模型/主题的材料

不要保留：

- 顺口评价
- 噱头式句子
- 无法独立成立的感叹
- 只在节目里轻轻带过、没有展开的点

## 各字段具体怎么写

### `summary`

`summary` 不是“本期聊了什么”，而是“本期的主判断是什么”。

要求：

- 2 到 3 句话
- 第一层说切入点
- 第二层说核心判断
- 如有必要，第三层说为什么重要

错误写法：

- `本期讨论了房票在各地流通的现象`
- `本期从豪宅市场切入，进行了深入分析`

正确方向：

- `本期围绕房票流通、折价出售和中介炒作展开，判断它不是楼市重新启动的信号，而是地方政府在现金吃紧时用票据代替部分现金、继续拖延出清的一种工具。`

### `topic.background`

写“节目是从什么现象切进来的，为什么这个现象值得讲”。

不要写百科背景。

要写：

- 这一期的现实入口
- 节目为什么拿它当样本
- 它和更大结构之间的关系

### `topic.conflicts`

写节目内部真正的矛盾，不是泛泛而谈的“支持与反对”。

要尽量写成两股力量或两种解释的对撞。

例如：

- `豪宅市场的坚挺，究竟是独立稀缺市场，还是整体楼市回暖信号`
- `房票流通是在激活真实需求，还是在延缓价格出清`

### `topic.boundaries`

写这一期没有做什么、不要误读成什么。

这个字段很重要，因为它能防止节目页显得全知全能。

常见边界包括：

- 不是法律审查
- 不是全面事实核查
- 不是对所有城市/行业的普遍结论
- 不是给个体投资建议

### `topic.mechanism`

这是节目页最值钱的字段。

最好写成因果链，而不是一个抽象句子。

推荐结构：

`A -> B -> C -> D`

例如：

- `断供增加 -> 法拍通道承载不足 -> 银行开辟直供房路径 -> 折价房源增加继续压低预期`

### `topic.extensions`

这里写的是“从节目内部还能自然延伸出的下一层问题”，不是整理者自我提醒。

正确方向：

- 继续追问
- 提出可讨论的结构问题
- 指向更大的长期议题

错误写法：

- `这一期可以作为后续整理地产主题的重要案例`
- `后续整理时可以沿着这个维度继续展开`

正确写法：

- `如果房票越来越普遍，未来楼市会不会出现票面价格稳定、真实成交价格不断场外折价的双轨市场`

### `viewpoints`

`viewpoints` 是最容易写差的部分。

最低要求：

- 每一条都必须是一个独立判断
- 标题和正文之间不能只是重复改写
- 不能把 summary 拆碎了重复说 6 次

好的 `viewpoints` 应该满足：

1. 这一条单独拿出来也成立
2. 它说的是节目里的一个具体判断，不是空话
3. 它和其他条之间不要重叠太多

推荐写法：

- 标题写判断
- 正文写证据、展开和后果

错误例子：

- 标题：`房价下跌可能不是一次到位，而是分阶段释放`
- 正文：`节目认为房价下跌可能不是一次到位，而是分阶段释放`

这类写法基本等于没写。

正确方向：

- 标题：`银行直供房不是抄底信号，而更像第一轮出清已经开始的标记`
- 正文：解释为什么节目会这么判断、后面还会看哪些信号、这对判断市场意味着什么

### `extensions`

根节点的 `extensions` 要写成“用户读完这期后自然会继续追问什么”。

不要写：

- `这一期适合与 XX 联读`
- `这一期可作为后续整理的入口`
- `整理时可继续沿着 XX 展开`

要写：

- 真正的问题
- 下一层判断
- 节目没有完全讲透、但已经被打开的方向

## 节点层怎么补

### 总原则

不是为了让“关联概念 / 关联模型 / 关联人物 / 关联主题”显示出来才去补节点。

而是因为：

- 这个节点真的能跨节目复用
- 它不是只服务这一集
- 它值得作为独立入口存在

### 概念怎么抽

概念比模型更小，回答的是：

- 这里到底出现了什么现象？

判断标准：

- 是否能用一句话定义
- 是否可能在别的节目再次出现
- 是否比节目标题更稳定

例如：

- `价格发现延迟`
- `隐形债务`
- `公司身份迁移`
- `语言工具化`

### 模型怎么抽

模型比概念更高一层，回答的是：

- 这些现象背后反复出现的机制是什么？

模型应该比概念更少、更稳。

优先复用已有模型：

- `路径依赖`
- `委托代理问题`
- `反馈回路`

如果已有模型不够，再新增新的中层模型。

例如：

- `资产负债表维持`
- `风险转移链条`
- `制度适配`

### 人物怎么补

只有人物真的构成该期的重要角色时才补。

不要因为 transcript 里提了一次名字就强行建人。

适合补人物的情况：

- 人是这期判断的关键载体
- 这个人在多期节目中重复出现
- 用户点进去确实会想知道“他是谁、为什么重要”

### 主题怎么补

主题是跨多期的议题层，不是标签升级版。

一个主题应该回答：

- 这些节目为什么应该被放到一起看

例如：

- `房地产去杠杆与风险出清`
- `价格托底与账面维稳`
- `AI创业、资本主场与公司身份`
- `小国平衡与地缘生存`

## 低质量写法黑名单

以下内容默认视为低质量，整理时要主动删掉或重写：

### 1. 整理者口吻

不要写：

- `这一期适合作为后续整理的入口`
- `后续可以沿着这个维度继续展开`
- `整理类似节目时可参考`

用户看到节目页，不应该读到整理后台备注。

### 2. 废话摘要

不要写：

- `本期讨论了……并进行了深入分析`
- `节目围绕……展开探讨`

必须把判断写出来。

### 3. 重复型观点

不要让 6 条 `viewpoints` 都在说同一件事。

如果标题和正文只是互相翻译，说明这条观点还没写开。

### 4. 空链接

- 空的 `people / themes / concepts / models / relatedEpisodes` 不要硬挂
- 但如果节目已经足够成熟，就应该主动补节点，而不是长期留空

### 5. 只显示不成体系

目标不是“页面上有东西”，而是：

- 节目能读
- 节点能点
- 节点之间能互相回跳

## 这轮实际采用的做法

当前高质量整理采用的是下面这套顺序：

1. 先读 transcript，抓主判断
2. 再写 `summary`
3. 再写 `topic.background / conflicts / boundaries / mechanism`
4. 再写彼此不重复的 `viewpoints`
5. 再写真正面向读者的 `extensions`
6. 再抽 `concepts / models / people / themes`
7. 最后补 `relatedEpisodes`

也就是说：

- 节目内容优先
- 节点层第二
- 页面显示第三

不要反过来做。

### 第四步：补稳定节点

如果节目里出现新的稳定内容，再补：

- 新概念：`content/concepts/*.json`
- 新模型：`content/models/*.json`
- 新人物：`content/people/*.json`
- 新主题：`content/themes/*.json`
- 新关键词：`content/keywords/*.json`

### 第五步：回填节目链接

把这些节点的 ID 回填到节目 JSON 里：

- `people`
- `themes`
- `concepts`
- `models`
- `relatedEpisodes`

### 第六步：重建和预览

```bash
npm run build
npm run serve
```

打开：

`http://127.0.0.1:4310/`

## 推荐工作顺序

每整理一集，按下面顺序走：

1. 先让节目页可读
2. 再抽概念
3. 再挂思想模型
4. 再补人物/主题/关键词
5. 最后回到网页检查跳转链路

不要一开始就试图把所有层全部写满。先保证节目条目清楚，再逐步向稳定节点沉淀。

## 命名规则

### 节目

- 文件名固定：`EPxxx.json`

### 概念 / 模型 / 人物 / 主题 / 关键词

- 文件名统一用 slug
- `id` 与文件名保持一致
- 这个规则不只适用于人物，概念 / 模型 / 主题 / 关键词也一样遵守
- 抽象概念、机制、模型、议题主题：优先用英文语义 slug，例如 `cost-shifting`
- 中文世界的专有名词：用拼音 slug，例如中国人物、中文品牌、中文公司、中文政策名、中文事件名
- 外国专有名词：用英文常用名 slug，例如外国人物、外国公司、外国机构、外国地名、外国事件
- 不要把“中国专有名词一律翻成英文”，也不要把“外国专有名词再转写成中文拼音”
- 旧错误 slug 不继续扩散，必要时放进 `aliases`

例如：

- `cost-shifting.json`
- `authority-misinfluence.json`
- `jia-guolong.json`
- `wahaha.json`
- `donald-trump.json`
- `sanae-takaichi.json`

补充判断：

- 如果节点本质上是“抽象机制”，即使讨论中国语境，也优先用英文语义 slug
- 如果节点本质上是“具体专有名词”，就按中文专名用拼音、外国专名用英文
- 如果历史上已经有旧 slug，先保留兼容到 `aliases`，不要再新造第三种写法

## GitHub 与上线发布标准

这个项目现在不是“本地改完就算上线”，而是 GitHub 驱动 Cloudflare 部署。

### 当前发布链路

- GitHub 仓库：`yinfluence/main`
- 生产分支：`main`
- Cloudflare Workers Builds 读取仓库里的 `docs/`

所以发布的关键不是只改本地文件，而是：

1. 本地把内容和页面改好
2. 如果这次准备上传 GitHub，先同步更新 `网页日志`
3. 执行 `npm run build`
4. 确认 `docs/` 已同步更新
5. `git add`
6. `git commit`
7. `git push` 到 `yinfluence/main`

如果不 `commit + push`，GitHub 不会更新，Cloudflare 也不会更新。

### 发布前最低动作

```bash
npm run build
git add -A
git commit -m "<符合 Lore 协议的提交信息>"
git push yinfluence-origin main
```

发布前额外硬规则：

- 只要这次要上传 GitHub，就必须先更新 `网页日志`
- `网页日志` 不需要写成长文，但至少要按时间列出这次改了哪些页面、交互或结构
- 不允许出现“代码已经 push 了，但网页日志还停在旧状态”

### 发布前必须检查

1. `网页日志` 已覆盖本次上传涉及的主要改动
2. `npm run build` 成功
3. `npm run test:ui` 成功，且桌面版与移动版都通过
4. `docs/index.html` 与当前构建产物一致
5. 本地关键页面已验证
6. GitHub 远端 `main` 已更新到最新 commit

### 如果 GitHub 更新了，但网站没更新

优先检查：

1. Cloudflare 有没有拉到新的 commit
2. Cloudflare Build 是否真的部署 `docs/`
3. 是否需要清缓存
4. 线上返回的 `index.html` 是否真的已经换成最新结构

硬检查方法：

1. 先核对 GitHub 远端 `main` 的最新 commit
2. 再直接请求线上首页 HTML
3. 看线上 HTML 里是否已经出现本次改动的关键标记

例如如果本次改动包含：

- `brand-lockup`
- `brand-avatar-link`
- `desktop-menu-button`

而线上 HTML 仍然没有这些标记，就说明：

- 不是 CSS 没生效
- 不是图片没加载
- 而是 Cloudflare 生产环境还在 serving 旧 deployment / 旧构建

也就是说：

- GitHub 新了，不代表线上一定新了
- 必须以线上实际返回的 HTML 为准，而不是只看仓库

### Cloudflare 本地直推的阻塞说明

如果要用 `wrangler deploy` 直接发布，当前环境必须满足以下任一条件：

- 本机已经 `wrangler login`
- 或已经设置 `CLOUDFLARE_API_TOKEN`

否则会出现：

- `You are not authenticated`
- 或 `set a CLOUDFLARE_API_TOKEN`

这类错误

一旦出现这类错误，说明当前终端没有直接控制 Cloudflare 的权限。
这时能做的事情是：

1. 先确保 GitHub 上的 `docs/` 已经是最新
2. 再去 Cloudflare 后台确认：
   - 绑定仓库是不是 `yinfluence/main`
   - production deployment 对应的 commit 是不是最新
   - output directory 是不是 `docs`

也就是说：

- GitHub 没更新：先 `push`
- GitHub 更新了但线上没变：查 Cloudflare deployment / cache

### 新终端执行原则

新终端如果完成了站点改动，默认不要停在“本地已经好了”。

必须继续做完：

1. `npm run build`
2. `git commit`
3. `git push yinfluence-origin main`

只有远端更新，才算真正进入部署链路。
