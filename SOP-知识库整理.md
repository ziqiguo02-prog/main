# 颖响力知识库 SOP

## 目标

这套 SOP 的目标不是“单篇写一篇文稿”，而是把每一期节目拆进一个可持续扩展的网页知识库里。最终产物包含两层：

- 内容层：节目、概念、思想模型、人物、主题、关键词
- 展示层：本地静态网页，`localhost` 可打开

你以后打开一个新终端，只需要传入 `.srt` 路径，就能先把新节目自动收录到这套知识库结构中，再继续补内容。

## 内容设计原则

### 1. 节目是入口，不是终点

每个节目 JSON 负责承接当期内容，但真正长期可复用的是：

- 概念
- 思想模型
- 人物
- 主题
- 关键词

节目要不断把内容往这些稳定节点上沉淀。

### 2. 信息、概念、思想模型是点线面关系

- 信息：字幕里的具体材料、表达、细节
- 概念：比思想模型小，概括一个现象
- 思想模型：多个概念按机制或因果关系组织成更高层框架

### 3. 思想模型优先复用 GV 现有模型

如果 GV 已经有贴合模型，优先复用模型名，再结合本期解释。
只有当 GV 现有模型明显不贴合时，才新增模型。

## 节目条目字段规范

每个节目 JSON 至少包含：

- `id`
- `title`
- `summary`
- `tags`
- `people`
- `themes`
- `concepts`
- `models`
- `relatedEpisodes`
- `topic`
- `viewpoints`
- `extensions`

其中：

- `topic`
  必须按这套结构写：
  - `background`
  - `conflicts`
  - `boundaries`
  - `mechanism`
  - `extensions`
- `viewpoints`
  必须写成分小标题的观点数组

## 六类内容怎么写

### 节目 `content/episodes/*.json`

负责承接本期的完整结构化整理，是网页详情页的主来源。

### 概念 `content/concepts/*.json`

每个概念都要能单独成立，最少要有：

- `name`
- `summary`
- `definition`
- `importance`
- `episodes`

### 思想模型 `content/models/*.json`

每个思想模型最少要有：

- `name`
- `summary`
- `definition`
- `application`
- `sourcePath`
- `episodes`

### 人物 `content/people/*.json`

人物不是简单标签，必须有最小介绍：

- `summary`
- `description`
- `episodes`

要求：

- `summary` 先回答“这个人是谁”
- `description` 先补这个人的背景，再写“在这些节目里主要怎么被提及”
- 不能只写标签式名字，必须有一句到一段可直接展示在网页上的简单背景介绍
- 不要让 `description` 一上来就是：
  - `在颖响力知识库里……`
  - `为什么重要……`
- 人物页的优先顺序是：
  1. 这个人是谁
  2. 他/她在这些节目里主要承接什么讨论线

### 主题 `content/themes/*.json`

主题用于把多个节目按议题聚合，最少要有：

- `summary`
- `description`
- `episodes`

### 关键词 `content/keywords/*.json`

关键词层用于承接品牌、人物、产品、公司、实体等更具体的检索入口，类似 GV 的公司层。

最少要有：

- `summary`
- `description`
- `aliases`
- `episodes`

要求：

- `summary` 先说明这个关键词是什么类型的对象，例如人物、公司、品牌、产品、实体入口
- `description` 必须先写“它是什么 / 做什么”，再写“它为什么在颖响力知识库里重要”

例如：

- 人物关键词：先写这个人是谁，再写其在专题里的位置
- 公司关键词：先写这家公司做什么，再写它在节目里代表的结构问题

## 关键词整理规则

关键词层的目标不是“把字幕里所有名词都做成平级入口”，而是优先形成稳定、好用的专题入口。

### 1. 优先保留上位专题，不滥开平级同义入口

- 优先保留真正能承接节目群的专题词，例如：
  - `娃哈哈`
  - `新能源汽车`
  - `西贝`
- 不要把明显属于同一簇的近义词、口语词、下位表达全部做成平级专题。

例如：

- `新能源 / 新能源汽车 / 电车`
  应优先收束到 `新能源汽车`
- `宗馥莉 / 宗庆后 / 娃哈哈`
  如果当前讨论主线是娃哈哈家族与公司问题，应优先收束到 `娃哈哈`
- `杂粮汽车 / 小米汽车`
  应收束到正式入口 `小米汽车`

### 2. 别名和口语词放到 `aliases`

- 如果一个词只是别称、口语、戏称、缩写，不要再单独做一个平级关键词页
- 把它放进主关键词的 `aliases`

例如：

- `新能源`、`电车` 作为 `新能源汽车` 的 `aliases`
- `哇哈哈`、`宗馥莉`、`宗庆后` 作为 `娃哈哈` 的 `aliases`
- `杂粮汽车` 作为 `小米汽车` 的 `aliases`

### 3. 人物是否留在关键词层，取决于是否是独立专题

- 如果人物只是服务于某个公司、品牌、家族或事件主线，优先并入上位关键词，不单独占入口
- 如果人物本身在多期节目里已经形成独立讨论线，再考虑保留为独立关键词或转入 `people`

判断标准：

- 问题核心是不是“这个人本身”
- 还是“这个人只是某个专题里的关键角色”

### 4. 上位词与下位词不要乱并列

- `汽车` 可以是上位词
- `新能源汽车` 是更具体的行业/产品专题
- `电车` 是更口语化、消费语境更强的表达

整理时优先选择最能代表该节目主线的那一个，不要同时堆多个高度重叠的泛词。

### 5. 单次出现的长尾词先不在关键词索引页展示

- 只出现 `1` 次的词可以保留在数据层，便于未来累计
- 但 `#/keywords` 列表页默认不展示
- 关键词索引页优先展示至少出现 `2` 次、已经形成节目群的词

### 6. 自动生成关键词文案不要写废话

- 不要写：
  - `XX相关关键词，聚合知识库里提到这个主题的N期节目`
- 应该写成一句真正有信息量的专题介绍

例如：

- `围绕房价的节目群入口，用来串联地产周期、政策压力和市场预期的相关讨论。`
- `围绕新加坡的节目群入口，用来串联其地缘位置、治理模式和区域博弈中的角色。`

### 7. 新整理节目时，先问自己这几个问题

1. 这个标签是不是已经属于现有专题？
2. 它是主入口，还是别名？
3. 它是独立专题，还是上位专题里的子人物 / 子问题？
4. 它现在只出现一次，是否值得先留在数据层、暂不放进列表页？

## 网页交互规范

### 返回按钮

- 详情页保留 `返回前一页`
- 同级索引返回按钮保留：
  - `返回概念页`
  - `返回模型页`
  - `返回人物页`
  - `返回主题页`
  - `返回关键词页`
- `返回前一页` 可以保留箭头
- `返回概念页` 这类按钮不带箭头，文案更干净
- `返回首页` 与 `返回概念页` 这类按钮使用同样的浅橙色样式

### hover 反馈

- 返回按钮需要有轻微 hover 变色
- 折叠项标题需要有轻微 hover 变色
- 可点击卡片、列表项、关联卡片都需要有轻微 hover 变色
- 原则是“让人感知得到可点击”，但不要做成厚重按钮感

### 左侧导航规则

- 左侧导航以“查找优先”为目标，不堆太长列表
- `关键词` 默认只显示 `3` 个
- `核心概念` 默认只显示 `5` 个
- `思想模型` 默认只显示 `5` 个
- 每个分组下方保留 `更多` 入口，跳转完整索引页
- `已整理节目` 不再放在左侧导航中

### 关键词搜索规则

- 左侧保留 `搜索关键词` 输入框
- 输入框下方默认显示 `推荐关键词`
- 推荐关键词默认不超过 `3` 个
- 开始输入后，标题切换为 `匹配结果`
- 左侧搜索不只搜关键词，也搜节目
- 输入 `31`、`31集`、`EP31`、`三十一集` 时，应能直接找到 `EP031`
- 按回车时优先跳到第一个精确命中的节目或关键词页

### 索引页排序与折叠规则

- `#/episodes` 的节目列表与相关节目列表默认按集数从大到小排序
  - 最新节目放在最前面
- `#/keywords` 的关键词专题卡片底部信息要固定在卡片底边，不允许高低错位
- `#/keywords` 只显示至少出现 `2` 次的关键词
- `#/concepts`、`#/models`、`#/people`、`#/themes`
  - 只显示被引用超过 `1` 次的条目
  - 默认按引用率从高到低排序
  - 按类别分组
  - 每个类别先显示前 `3` 个
  - 其余条目折叠到 `更多`
- `#/people`
  - 人物分类优先使用人工映射，其次才允许规则兜底
  - 像 `高市早苗`、`特朗普` 这类边界人物，不能只靠关键词猜分类
- `#/models`
  - 分类要比其他页更细，优先按机制层区分，而不是只分几个大筐

### 节目详情页展开规则

- `话题` 下各项使用折叠结构
- `核心观点` 下各项使用折叠结构
- `延展问题` 不折叠，直接展开
- `关联人物` 不折叠，直接展开
- `关联主题` 不折叠，直接展开
- `关联节目` 不折叠，直接展开

### 关联节目展示规则

- 关联节目不要用横向标签
- 关联节目统一使用纵向列表
- 每条列表至少显示：
  - 节目编号
  - 标题
  - 一行摘要

### 脚本会自动做什么

脚本：

`scripts/ingest-srt.mjs`

自动完成以下动作：

1. 检查文件名是否符合：
   `[EPxxx】 标题.srt`
2. 如果文件不在 `bilibili/raw/`，自动复制进去
3. 把字幕去掉时间轴，抽成纯文本 transcript
4. 在 `workbench/EPxxx/` 下生成：
   - `EPxxx.transcript.txt`
   - `EPxxx.intake.md`
5. 在 `content/episodes/` 下生成节目草稿 JSON
6. 自动执行 `npm run build`

批量脚本 `bootstrap:all` 会做同样的事，但作用对象是 `raw/` 中所有尚未生成节目 JSON 的 EP 文件。

### 脚本不会自动做什么

脚本不会自动替你完成高质量知识整理，它只负责：

- 收录
- 转写
- 搭草稿
- 重建网页

批量导入完成后，这些新节目会以 `draft` 状态进入知识库，而不是伪装成已经精修完成的条目。

你还需要手动补：

- 节目摘要
- 话题结构
- 核心观点
- 概念
- 思想模型
- 人物/主题/关键词链接

## 节目内容怎么做

这一部分是整个知识库最重要的工作面。不要把节目条目理解成“字幕摘要”，而要把它理解成：

- 这一期到底在解释什么
- 它的主判断是什么
- 它把哪些现象连成了一条因果链
- 哪些节点值得被沉淀成长期可复用的概念、模型、人物、主题

### 基本原则

- 先抓“节目真正想说什么”，再写字段
- 不要按字幕顺序机械抄录
- 不要把一堆正确废话拆成 bullets
- 不要写成整理者备注
- 要写成“用户点开就能读懂这一期在讲什么”

### 节目页的黄金标准

当前节目条目的参考标准是：

- `EP001`

它的特点不是字多，而是结构完整：

- 有明确主判断
- `topic` 不是空架子
- `viewpoints` 每条都有独立判断价值
- `concepts / models / people / themes / relatedEpisodes` 都不是装饰
- 节目本身能反向推动节点层变厚

整理新节目时，默认以 `EP001` 的完成度为目标，而不是以“能显示出来”为目标。

## 如何从 transcript 提炼节目主线

### 第一步：先找主判断

先回答一句话：

- 这期最想说的核心判断是什么？

如果这句答不出来，就不要急着填 JSON。

正确做法：

- 先把节目里的判断压缩成一句话
- 这句话必须带方向感，而不是中性复述

例如：

- 不要写：`节目讨论了银行直供房、法拍房和房价问题`
- 要写：`节目把银行直供房视为法拍通道装不下风险后，银行为回收流动性开辟的新处置路径，而不是楼市见底信号`

### 第二步：再找它的论证链

问自己：

1. 它拿什么现象开头
2. 它把这些现象解释成什么机制
3. 它最后落到什么判断或建议

通常可以整理成：

- 起点现象
- 核心矛盾
- 解释机制
- 最终落点

### 第三步：只保留节目真正强调的部分

不是 transcript 里出现过的每句话都值得进结构化条目。

优先保留：

- 反复出现的判断
- 串联多个细节的解释框架
- 节目最后明确落下来的结论
- 能沉淀为概念/模型/主题的材料

不要保留：

- 顺口评价
- 噱头式句子
- 无法独立成立的感叹
- 只在节目里轻轻带过、没有展开的点

## 各字段具体怎么写

### `summary`

`summary` 不是“本期聊了什么”，而是“本期的主判断是什么”。

要求：

- 2 到 3 句话
- 第一层说切入点
- 第二层说核心判断
- 如有必要，第三层说为什么重要

错误写法：

- `本期讨论了房票在各地流通的现象`
- `本期从豪宅市场切入，进行了深入分析`

正确方向：

- `本期围绕房票流通、折价出售和中介炒作展开，判断它不是楼市重新启动的信号，而是地方政府在现金吃紧时用票据代替部分现金、继续拖延出清的一种工具。`

### `topic.background`

写“节目是从什么现象切进来的，为什么这个现象值得讲”。

不要写百科背景。

要写：

- 这一期的现实入口
- 节目为什么拿它当样本
- 它和更大结构之间的关系

### `topic.conflicts`

写节目内部真正的矛盾，不是泛泛而谈的“支持与反对”。

要尽量写成两股力量或两种解释的对撞。

例如：

- `豪宅市场的坚挺，究竟是独立稀缺市场，还是整体楼市回暖信号`
- `房票流通是在激活真实需求，还是在延缓价格出清`

### `topic.boundaries`

写这一期没有做什么、不要误读成什么。

这个字段很重要，因为它能防止节目页显得全知全能。

常见边界包括：

- 不是法律审查
- 不是全面事实核查
- 不是对所有城市/行业的普遍结论
- 不是给个体投资建议

### `topic.mechanism`

这是节目页最值钱的字段。

最好写成因果链，而不是一个抽象句子。

推荐结构：

`A -> B -> C -> D`

例如：

- `断供增加 -> 法拍通道承载不足 -> 银行开辟直供房路径 -> 折价房源增加继续压低预期`

### `topic.extensions`

这里写的是“从节目内部还能自然延伸出的下一层问题”，不是整理者自我提醒。

正确方向：

- 继续追问
- 提出可讨论的结构问题
- 指向更大的长期议题

错误写法：

- `这一期可以作为后续整理地产主题的重要案例`
- `后续整理时可以沿着这个维度继续展开`

正确写法：

- `如果房票越来越普遍，未来楼市会不会出现票面价格稳定、真实成交价格不断场外折价的双轨市场`

### `viewpoints`

`viewpoints` 是最容易写差的部分。

最低要求：

- 每一条都必须是一个独立判断
- 标题和正文之间不能只是重复改写
- 不能把 summary 拆碎了重复说 6 次

好的 `viewpoints` 应该满足：

1. 这一条单独拿出来也成立
2. 它说的是节目里的一个具体判断，不是空话
3. 它和其他条之间不要重叠太多

推荐写法：

- 标题写判断
- 正文写证据、展开和后果

错误例子：

- 标题：`房价下跌可能不是一次到位，而是分阶段释放`
- 正文：`节目认为房价下跌可能不是一次到位，而是分阶段释放`

这类写法基本等于没写。

正确方向：

- 标题：`银行直供房不是抄底信号，而更像第一轮出清已经开始的标记`
- 正文：解释为什么节目会这么判断、后面还会看哪些信号、这对判断市场意味着什么

### `extensions`

根节点的 `extensions` 要写成“用户读完这期后自然会继续追问什么”。

不要写：

- `这一期适合与 XX 联读`
- `这一期可作为后续整理的入口`
- `整理时可继续沿着 XX 展开`

要写：

- 真正的问题
- 下一层判断
- 节目没有完全讲透、但已经被打开的方向

## 节点层怎么补

### 总原则

不是为了让“关联概念 / 关联模型 / 关联人物 / 关联主题”显示出来才去补节点。

而是因为：

- 这个节点真的能跨节目复用
- 它不是只服务这一集
- 它值得作为独立入口存在

### 概念怎么抽

概念比模型更小，回答的是：

- 这里到底出现了什么现象？

判断标准：

- 是否能用一句话定义
- 是否可能在别的节目再次出现
- 是否比节目标题更稳定

例如：

- `价格发现延迟`
- `隐形债务`
- `公司身份迁移`
- `语言工具化`

### 模型怎么抽

模型比概念更高一层，回答的是：

- 这些现象背后反复出现的机制是什么？

模型应该比概念更少、更稳。

优先复用已有模型：

- `路径依赖`
- `委托代理问题`
- `反馈回路`

如果已有模型不够，再新增新的中层模型。

例如：

- `资产负债表维持`
- `风险转移链条`
- `制度适配`

### 人物怎么补

只有人物真的构成该期的重要角色时才补。

不要因为 transcript 里提了一次名字就强行建人。

适合补人物的情况：

- 人是这期判断的关键载体
- 这个人在多期节目中重复出现
- 用户点进去确实会想知道“他是谁、为什么重要”

### 主题怎么补

主题是跨多期的议题层，不是标签升级版。

一个主题应该回答：

- 这些节目为什么应该被放到一起看

例如：

- `房地产去杠杆与风险出清`
- `价格托底与账面维稳`
- `AI创业、资本主场与公司身份`
- `小国平衡与地缘生存`

## 低质量写法黑名单

以下内容默认视为低质量，整理时要主动删掉或重写：

### 1. 整理者口吻

不要写：

- `这一期适合作为后续整理的入口`
- `后续可以沿着这个维度继续展开`
- `整理类似节目时可参考`

用户看到节目页，不应该读到整理后台备注。

### 2. 废话摘要

不要写：

- `本期讨论了……并进行了深入分析`
- `节目围绕……展开探讨`

必须把判断写出来。

### 3. 重复型观点

不要让 6 条 `viewpoints` 都在说同一件事。

如果标题和正文只是互相翻译，说明这条观点还没写开。

### 4. 空链接

- 空的 `people / themes / concepts / models / relatedEpisodes` 不要硬挂
- 但如果节目已经足够成熟，就应该主动补节点，而不是长期留空

### 5. 只显示不成体系

目标不是“页面上有东西”，而是：

- 节目能读
- 节点能点
- 节点之间能互相回跳

## 这轮实际采用的做法

当前高质量整理采用的是下面这套顺序：

1. 先读 transcript，抓主判断
2. 再写 `summary`
3. 再写 `topic.background / conflicts / boundaries / mechanism`
4. 再写彼此不重复的 `viewpoints`
5. 再写真正面向读者的 `extensions`
6. 再抽 `concepts / models / people / themes`
7. 最后补 `relatedEpisodes`

也就是说：

- 节目内容优先
- 节点层第二
- 页面显示第三

不要反过来做。

### 第四步：补稳定节点

如果节目里出现新的稳定内容，再补：

- 新概念：`content/concepts/*.json`
- 新模型：`content/models/*.json`
- 新人物：`content/people/*.json`
- 新主题：`content/themes/*.json`
- 新关键词：`content/keywords/*.json`

### 第五步：回填节目链接

把这些节点的 ID 回填到节目 JSON 里：

- `people`
- `themes`
- `concepts`
- `models`
- `relatedEpisodes`

### 第六步：重建和预览

```bash
npm run build
npm run serve
```

打开：

`http://127.0.0.1:4310/`

## 推荐工作顺序

每整理一集，按下面顺序走：

1. 先让节目页可读
2. 再抽概念
3. 再挂思想模型
4. 再补人物/主题/关键词
5. 最后回到网页检查跳转链路

不要一开始就试图把所有层全部写满。先保证节目条目清楚，再逐步向稳定节点沉淀。

## 命名规则

### 节目

- 文件名固定：`EPxxx.json`

### 概念 / 模型 / 人物 / 主题 / 关键词

- 文件名用英文 slug
- `id` 与文件名保持一致

例如：

- `cost-shifting.json`
- `authority-misinfluence.json`
- `jia-guolong.json`

## 验收标准

一条新节目整理完成，至少满足：

1. 首页或节目索引里能点进去
2. 节目页有完整 `summary / topic / viewpoints`
3. 至少链接到一组稳定节点：
   - 概念
   - 思想模型
   - 人物 / 主题 / 关键词
4. 页面能从节目跳到概念，再跳回节目
5. `npm run build` 成功

## 当前状态约定

- `status: "curated"` 表示这期已经做了相对完整的结构化整理
- `status: "draft"` 表示这期已经进入知识库，但仍然是自动生成的草稿，需要继续人工补内容
- 首页统计中的：
  - `curatedEpisodes` = 已精修节目数
  - `draftEpisodes` = 已导入但待整理节目数
