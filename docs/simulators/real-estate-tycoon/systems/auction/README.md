# 土拍系统

放土拍相关代码：

- `competitors.js`：圈内竞买人名册、生命周期、资金变化和成交后结果记录。
- `city-map.js`：城市规划底图，负责道路、水系、轨道、工业区、农业区、街区和区域标签。
- `map-view.js`：土拍地图节点渲染，可拍地、自有地、别人拍走的灰色地块都在这里。
- `app.js` 里的 `cityPlanningSnapshot()` 暂时负责把回合、区域热度和区域房价转换成地图阶段：大片农地、低效工业、预留新区、升温走廊、农转用/工改商。
- 地图地块生成、区域房价波动、已成交地块归属。
- 拍前动作：圈内沟通、联合竞标、进入拍卖；估价信息并入正式拍卖面板，不再单独占一个按钮。
- 拍前圈内沟通不在土拍阶段送钱，也不增加亲密度；关系增长回到关系页的走动/送礼动作。沟通结果按亲密度和谈法分成“反向试探 / 套话 / 半句口径 / 具体口径”，只有具体口径才会真正改变竞拍牌桌。
- 拍卖轮次：跟价、退出、竞争对手举牌、联合方叫停。
- 联合竞标里，只有真正出资的圈内人能带共同资金和价格叫停权；政府/银行窗口只提供协调或口径，不出资，也不能替玩家终止举牌。
- 成交结果：玩家拿地、别人拍走、灰色地块显示。
- 竞争对手：每个规模层有 4-5 个圈内人物假名，带资金、性格、偏好和生命周期；他们会扩张、保守、出清或退场。

约束：土拍不能绕过资金校验。没有保证金或共同资金不足时，只能退出或先去融资/找联合方。

## 下一步土拍计划

1. 竞拍必须是多轮往返：玩家加价后对手可以加价，但不能因为对手临时领跑就自动结束；只有玩家主动退出、资金不足、联合方叫停，或玩家领跑到落槌条件时才结算。
2. 圈内沟通重做成“分类 -> 具体人 -> 亲密度 -> 谈判选项”：本地老板、政府窗口、银行窗口各自有联系人；亲密度可为负，负数代表仇敌，仇敌和强硬有钱的老板会反向抬价或拒绝谈。
3. 移动端土拍优先：进入经营回合默认显示土拍城市地图；选中地块后隐藏最近成交、重复地块详情和状态栏干扰，只保留地图和紧凑拍前按钮。
4. 土地生命周期：开局以农田、低效工业、边缘地为主；随着回合、交通、规划、产业和政府口径变化，部分地块转为新区、工业区、商业/住宅可开发地，地价随之跳升。
5. 持地策略：玩家拿地后 5 回合内要决定开发、继续捂地、抵押融资或转让；捂地能吃规划升值，但会占用资金、增加政策和闲置风险。
6. 地价模型：不做匀速上涨，按区位、配套、规划转换、城市周期、供地节奏和老城更新成本分区波动；核心区早期贵但后期涨得慢，农田/边缘地便宜但靠规划兑现。
7. 专注流程：进入圈内沟通、联合竞标、正式拍卖后隐藏大地图和老板侧栏；左侧只放一字分类，右侧上方横滑具体人，下面显示可谈方案或竞拍动作。
8. 现场感：正式拍卖时对手卡片负责表现“谁加价/谁停手/他说什么”，日志只保留事实，不把台词堆在价格旁边。

## 因果事件计划

突发事件不能只从随机事件池里抽。系统要记录“选择留下了什么因”，然后在后续回合优先触发对应事件。

- 选择记录：每次选择保存回合、事件、选项、人物/对象、机制标签和后续原因。
- 队列记录：`enqueueEvent` 不只存事件 id，也存前因；事件触发时标题下显示“前因：第几回合你做了什么”。
- 强因果：偷工减料、压工程款、挪预售款、灰线融资、承诺干股、联合体让出控制权、恶意营销、延期交付等，必须优先排进相关事件。
- 弱因果：周期下行、银行收紧、政策检查、行业舆论等可以随机出现，但权重仍要受玩家现金、项目、关系和风险账本影响。
- 人物因果：和某个老板/官员/银行经理谈过的条件，会影响之后他是否帮忙、反向抬价、叫停联合竞标、要求补材料或把风险推回玩家。
